Compatibility Between PO3 and NG Forks (Code-Level)

The two CommonLibSSE forks supported by this template differ primarily in how they expose internal runtime data:

  • NG Fork: Encapsulates internal data behind accessor functions, such as GetRuntimeData().
  • PO3 Fork: Exposes data directly via public members like renderer->data.

Because of these differences, plugin code must use conditional compilation to access data correctly depending on the selected fork. For example:


#if defined(SKYRIM_SUPPORT_NG)
    auto& runtimeData = renderer->GetRuntimeData();
    // Access runtimeData fields here
#else
    auto& data = renderer->data;
    // Access data fields here
#endif

This pattern is common across the API surface and ensures your code compiles and runs correctly with either fork. The macro SKYRIM_SUPPORT_NG is defined automatically when building with the NG fork; otherwise, PO3 compatibility is assumed.
